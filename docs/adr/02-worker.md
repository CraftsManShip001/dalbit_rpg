# ADR 02 — Web Worker 격리 실행

## 상태
결정됨

## 컨텍스트

달빛약속 코드를 브라우저에서 실행해야 한다. 플레이어가 작성하는 코드는 신뢰할 수 없다. 실수로 무한루프를 작성하거나, 알고리즘 문제에서 느린 코드를 제출할 수 있다. 이 경우 메인 스레드가 블로킹되면 게임 전체가 멈춘다.

실행 환경의 안전성 확보가 필요했다.

## 선택지

**메인 스레드 직접 실행**
구현이 가장 단순하다. 무한루프 코드 실행 시 UI가 완전히 멈춘다. 게임 자체가 먹통이 되는 상황을 방지할 방법이 없다.

**Web Worker + 타임아웃**
별도 스레드에서 실행하므로 메인 스레드 블로킹이 없다. `worker.terminate()`로 강제 종료가 가능하다. 타임아웃을 설정해서 일정 시간 초과 시 Worker를 종료할 수 있다.

**iframe sandbox**
격리 실행이 가능하다. Worker보다 통신 구조가 복잡하다. `postMessage`/`addEventListener` 외에 `srcdoc`, `sandbox` 속성 등 추가 설정이 필요하다.

## 결정

**Web Worker + 매 실행마다 새 Worker 생성** 방식을 선택한다.

## 근거

플레이어가 실수로 무한루프를 작성하는 상황은 교육용 게임에서 충분히 예상 가능하다. 이때 게임이 먹통이 되면 사용자 경험이 완전히 무너진다. 메인 스레드 직접 실행은 이 리스크를 해결할 수 없어서 제외했다.

iframe은 통신 복잡도가 높고 Worker로 충분히 해결 가능한 문제라 선택하지 않았다.

Worker 재사용 방식을 검토했으나, `terminate()` 후 같은 Worker 인스턴스를 재사용할 수 없다. 타임아웃으로 종료된 Worker는 새로 생성해야 한다. 매 실행마다 새 Worker를 생성하면 이 문제가 없고, 성능보다 안전성을 우선하는 이 프로젝트의 맥락에 적합하다.

## 결과

antigravity 브라우저 테스트에서 무한루프 코드(`반복\n    1 보여주기`) 실행 시 5초 후 `{ error: 'TIMEOUT' }` 반환을 확인했다. 타임아웃 대기 중 UI 버튼 클릭이 즉각 반응하는 것도 확인했다.

추가 발견사항: 달빛약속 파서는 `반복` 블록 내부에 `반복 그만`이 없으면 실행 전에 `RUNTIME_ERROR`를 먼저 반환한다. 실제 타임아웃 유도 테스트를 위해 빈 반복문이 아닌 내용이 있는 무한루프 코드를 사용해야 한다.
